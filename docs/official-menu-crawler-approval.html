Official Menu Crawler And Review Workflow

This document summarizes the production-ready local workflow for discovering official venue menu sources, extracting candidate pint prices, reviewing them, and preparing pending price_reports rows. The validated path is dry-run by default and does not write to Supabase unless an explicit --write command is approved.

Workflow Commands

1. Discover candidate official menu sources

npm run discover:official-menu-sources -- --limit 200 --output scripts/official-menu-source-candidates-large.json

2. Build a reviewed seed file

npm run build:official-menu-seeds -- --input scripts/official-menu-source-candidates-large.json --output scripts/official-menu-seeds.large.json --min-score 8 --per-pub 2

3. Crawl sources in dry-run mode

npm run crawl:official-menus -- --file scripts/official-menu-seeds.large.json --output scripts/official-menu-crawl-results-large.json

4. Export review queue

npm run build:official-menu-review

5. Export suggested decisions

npm run build:official-menu-review -- --suggest-decisions --csv-output scripts/official-menu-review.suggested.csv --json-output scripts/official-menu-review.suggested.json

6. Build dry-run import plan

npm run import:official-menu-review

7. Optional write command after explicit approval

npm run import:official-menu-review -- --write

The write path uses SUPABASE_SERVICE_ROLE_KEY by default, checks existing official_menu reports before inserting, and only uses the anon public-insert path when --allow-anon-write is explicitly supplied.

Production Guardrails

  • All crawler, review, and importer scripts default to local/dry-run behavior.
  • No crawler/importer script updates canonical pubs.price; importer creates pending price_reports only after --write, and canonical price updates still go through the admin review path.
  • Official-menu provenance is preserved with submission_source, source_url, evidence_text, observed_at, raw_extraction, and extractor_version.
  • The importer prefers SUPABASE_SERVICE_ROLE_KEY for writes, requires an explicit --allow-anon-write flag for the public insert policy, and checks existing official-menu reports to avoid duplicates.
  • Generated artifacts are ignored by git so crawl output and OCR artifacts do not become source-controlled noise.
  • Internet usage is limited to fetching public venue/menu URLs; parsing, OCR, review, and import planning run locally.

Open-source Research Applied

Research did not find a complete open-source solution for reliable "restaurant URL to normalized pint price" extraction. The implemented workflow follows the recommended layered approach instead:

  • Use browser rendering for dynamic pages rather than relying only on raw HTML.
  • Use deterministic PDF text/table extraction before OCR.
  • Use OCR as a fallback for image and scanned PDF sources.
  • Keep LLM/service-based tools as later fallbacks, not the default production path.
  • Keep human review between extraction and any database write.

Known Limits

  • Many venues publish beer prices without explicitly saying "pint"; these remain needs_manual_review.
  • Some venue URLs 403, 404, timeout, or fail fetch; those are captured in artifacts but not retried automatically yet.
  • OCR can produce messy text; suspicious OCR rows are kept out of auto-approval suggestions.
  • The workflow currently imports pending reports, not direct canonical pub price updates.
  • Crawlee, Docling, OCRmyPDF, PaddleOCR, Firecrawl, and LLM-based extractors are not integrated yet.
